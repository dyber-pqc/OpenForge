# OpenForge Crypto Verification Suite

The crypto verification suite provides pre-silicon security analysis for cryptographic hardware implementations. It detects timing side channels, evaluates power leakage, validates post-quantum algorithm correctness, checks FIPS 140-3 compliance, and assesses fault injection resilience.

## Overview

The suite consists of six analysis engines, each targeting a different class of cryptographic vulnerability:

| Engine | Location | Purpose |
|--------|----------|---------|
| Constant-Time Verifier | `packages/crypto/src/openforge_crypto/constant_time.py` | Detect secret-dependent timing behavior |
| Side-Channel Simulator | `packages/crypto/src/openforge_crypto/side_channel.py` | Evaluate power leakage via statistical tests |
| Entropy Flow Analyzer | `packages/crypto/` | Verify entropy propagation paths |
| FIPS Compliance Checker | `packages/crypto/` | Automated FIPS 140-3 checks |
| NTT Validator | `packages/crypto/src/openforge_crypto/nist_vectors.py` | Post-quantum algorithm correctness |
| Fault Injection Simulator | `packages/crypto/` | Resilience against physical attacks |

Rust accelerated versions of the constant-time analyzer (`tools/openforge-ct/`), side-channel engine (`tools/openforge-sca/`), and entropy analyzer (`tools/openforge-entropy/`) provide native-speed processing for large designs.

## Constant-Time Analysis

**Module**: `openforge_crypto.constant_time`

The constant-time verifier detects timing side-channel vulnerabilities where secret-dependent data influences control flow or memory access patterns.

### Taint Propagation Algorithm

The analysis works in four phases:

1. **Graph construction**: Build a dataflow graph (DFG) where nodes are signals and edges represent data dependencies. Nodes are annotated with control-flow properties (branch condition, array index, variable-latency operation, loop bound).

2. **Secret annotation**: Mark input signals as `SECRET` or `PUBLIC` based on the project configuration (the `constant_time.secrets` and `constant_time.public` lists in `openforge.yaml`).

3. **Taint propagation**: Forward-propagate `SECRET` taint through the DFG until reaching a fixpoint. Any node reachable from a secret source inherits the `SECRET` taint level.

4. **Violation detection**: Check all tainted nodes for control-flow sensitivity. A violation is reported when a secret-tainted signal is used as:
   - **Branch condition** (`SECRET_BRANCH`): Mux select, if/case condition
   - **Array index** (`SECRET_INDEX`): Memory address (cache timing)
   - **Variable-latency operand** (`VARIABLE_LATENCY_OP`): Division, floating-point
   - **Loop bound** (`SECRET_LOOP_BOUND`): Iteration count

### Taint Path Tracing

When a violation is found, BFS backwards through the reverse DFG reconstructs the shortest path from the secret source to the violating node. This taint path is included in the report for debugging.

### SVA Generation

For each detected violation, the verifier can generate SystemVerilog Assertions that formally prove the absence of the timing channel when the violation is fixed:

```systemverilog
// Generated by OpenForge constant-time verifier
assert property (@(posedge clk) disable iff (!rst_n)
    !$changed(branch_sel) || !depends_on_secret(branch_sel)
) else $error("SECRET_BRANCH: branch_sel depends on secret data");
```

### Report Output

The `CTReport` includes:
- Module name and PASS/FAIL status
- Number of signals analyzed
- Number of branches analyzed
- Taint graph node count
- List of violations with kind, signal name, location, message, and taint path

## Side-Channel Simulation

**Module**: `openforge_crypto.side_channel`

The side-channel simulator evaluates power leakage using simulated power traces and statistical tests.

### Power Models

Two power models are supported:

**Hamming Weight (HW)**: Models instantaneous power consumption as proportional to the number of set bits in the processed value. Suitable for CMOS circuits where dynamic power dominates.

```
power(v) = popcount(v) + noise
```

**Hamming Distance (HD)**: Models power as proportional to the number of bit transitions between consecutive register states. More accurate for register-heavy designs.

```
power(v_prev, v_curr) = popcount(v_prev XOR v_curr) + noise
```

### TVLA Methodology

Test Vector Leakage Assessment (TVLA) uses Welch's t-test to detect information leakage:

1. **Trace collection**: Generate two groups of power traces:
   - **Fixed group**: Encrypt a fixed plaintext repeatedly
   - **Random group**: Encrypt random plaintexts

2. **Statistical test**: Compute the per-sample t-statistic:
   ```
   t[s] = (mean_fixed[s] - mean_random[s]) / sqrt(var_fixed[s]/n0 + var_random[s]/n1)
   ```

3. **Leakage detection**: Any sample point where `|t| > threshold` (default 4.5) indicates statistically significant leakage.

The `TVLAResult` reports the t-statistic vector, leaking sample indices, maximum t-value, and overall leakage verdict.

### CPA Attack Flow

Correlation Power Analysis validates that a design resists key recovery:

1. **Hypothesis generation**: For each of the 256 possible key byte values, compute the hypothetical intermediate value (AES SubBytes output) and its Hamming weight.

2. **Correlation computation**: Calculate Pearson correlation between the hypothetical power model and actual power traces at each sample point.

3. **Key ranking**: The key byte with the highest correlation is the attack's best guess. Success is determined by the margin between the best and second-best correlation values.

The `CPAResult` provides the full 256 x N correlation matrix, the best key byte guess, correlation strength, and success verdict.

### Configuration

```yaml
crypto_verification:
  side_channel:
    enabled: true
    power_model: hamming_distance   # hamming_weight or hamming_distance
    tvla_threshold: 4.5             # Absolute t-value threshold
    num_traces: 10000               # Number of traces to simulate
```

## Entropy Flow Analysis

The entropy flow analyzer verifies that random number generators and entropy sources correctly propagate randomness to all cryptographic consumers.

### Source/Sink Model

The analysis models entropy flow as a directed graph:

- **Sources**: Hardware random number generators (TRNGs), seed inputs, nonce generators
- **Sinks**: Key generation logic, mask generators, randomized countermeasures
- **Paths**: Dataflow connections between sources and sinks

### Reduction Detection

The analyzer detects entropy reduction operations that may compromise randomness:

- **Truncation**: Discarding bits from an entropy source
- **Deterministic mixing**: XOR with a constant or predictable value
- **Compression**: Many-to-one mappings that reduce entropy
- **Reuse**: Same random value used in multiple independent operations

### Configuration

```yaml
crypto_verification:
  entropy_analysis:
    sources: [trng_output, seed_input]
    sinks: [key_gen_rand, mask_gen_rand]
```

## FIPS 140-3 Compliance

Automated compliance checking against FIPS 140-3 requirements for cryptographic modules.

### Automated Checks

| Check | Description |
|-------|-------------|
| `key_zeroization` | Verify that all key material is actively overwritten (not just freed) when no longer needed |
| `self_test_coverage` | Ensure Known Answer Tests (KATs) cover all algorithm implementations |
| `error_handling` | Validate that cryptographic errors trigger appropriate state transitions |
| `rng_health` | Check that RNG output passes NIST SP 800-90B health tests |
| `integrity` | Verify module integrity checking at power-on |
| `pairwise` | Pairwise consistency tests for key generation |

### SVA Property Generation

For each FIPS check, the suite generates SVA properties that can be formally verified with SymbiYosys:

```systemverilog
// FIPS 140-3: Key zeroization
property p_key_zeroize;
    @(posedge clk) disable iff (!rst_n)
    $fell(key_valid) |-> ##[1:MAX_ZEROIZE_CYCLES] (key_reg == '0);
endproperty

assert property (p_key_zeroize)
    else $error("FIPS: key not zeroized within deadline");
```

### Report Format

FIPS compliance reports include:
- Target FIPS level (1-4)
- Per-check PASS/FAIL status with evidence
- Generated SVA properties for formal verification
- Remediation guidance for failures

### Configuration

```yaml
crypto_verification:
  fips_compliance:
    enabled: true
    level: "140-3 Level 2"
    checks:
      - key_zeroization
      - self_test_coverage
      - error_handling
      - rng_health
```

## NTT Validation

Validates Number Theoretic Transform implementations used in post-quantum lattice-based cryptography (FIPS 203 Kyber / FIPS 204 Dilithium).

### Reference Comparison

The validator compares the hardware NTT implementation against bit-exact reference models:

- **FIPS 203 (ML-KEM / Kyber)**: NTT over Z_q with q = 3329, n = 256
- **FIPS 204 (ML-DSA / Dilithium)**: NTT over Z_q with q = 8380417, n = 256

Reference vectors are generated from the official NIST specifications and stored in `packages/crypto/src/openforge_crypto/nist_vectors.py`.

### Butterfly Verification

Each NTT butterfly operation is verified independently:

1. **Forward butterfly**: `a' = a + w*b mod q`, `b' = a - w*b mod q`
2. **Inverse butterfly**: Reverse transform with inverse twiddle factors
3. **Modular arithmetic**: Verify that Montgomery/Barrett reduction produces correct results

### Exhaustive Testing

When `exhaustive: true` is set in the configuration, the validator tests all possible input combinations for small butterfly sizes. For full NTT, it uses NIST Known Answer Test vectors.

### Configuration

```yaml
crypto_verification:
  ntt_validation:
    standard: kyber          # kyber, dilithium, or custom
    exhaustive: false         # Full exhaustive testing (slow)
```

## Fault Injection Analysis

Simulates physical fault attacks and evaluates the design's resilience.

### Fault Models

| Model | Description | Parameters |
|-------|-------------|------------|
| **Glitch** | Clock glitch causing setup/hold violations | Duration, injection point |
| **Bit-flip** | Single or multi-bit flip in registers | Target register, bit mask |
| **Laser** | Spatially targeted fault (multiple adjacent bits) | Target region, radius |

### Differential Fault Analysis (DFA)

The simulator performs DFA on block ciphers:

1. Inject faults at specific rounds during encryption
2. Collect faulty and correct ciphertexts
3. Analyze differential pairs to determine if the secret key can be recovered
4. Report the minimum number of faults required for key extraction

### Resilience Scoring

Each design receives a fault resilience score based on:

- **Detection rate**: Percentage of injected faults that trigger error detection logic
- **Containment rate**: Percentage of detected faults that are properly handled (output suppressed)
- **Recovery rate**: Percentage of faults from which the module recovers to a safe state
- **Overall score**: Weighted combination (detection 40%, containment 35%, recovery 25%)

Scores are reported per fault model and as an aggregate.

## Formal Verification Integration

The crypto verification suite integrates with SymbiYosys for formal proof of security properties.

### Workflow

1. The crypto analysis engines identify potential vulnerabilities and generate SVA properties.
2. Properties are written to `.sv` files in the project's `formal/` directory.
3. A SymbiYosys job file (`.sby`) is generated with appropriate engine selection (SMTBMC for bounded model checking, ABC PDR for unbounded proofs).
4. The flow engine dispatches the formal verification job.
5. Results are parsed and merged back into the crypto verification report.

### Supported SymbiYosys Engines

| Engine | Strategy | Best For |
|--------|----------|----------|
| `smtbmc` | Bounded model checking (BMC) | Bug finding, bounded proofs |
| `btor` | BTOR2 witnesses | Counter-example generation |
| `aiger` | AIG-based model checking | Hardware-specific proofs |
| `abc` | PDR (Property Directed Reachability) | Unbounded proofs |

## SVA Crypto Property Library

OpenForge ships with a library of reusable SystemVerilog Assertion templates for common cryptographic security properties, located in `share/sva/crypto/`.

### Property Categories

| Category | Properties |
|----------|------------|
| **Constant-time** | No secret-dependent branches, no secret-dependent memory access, no variable-latency ops on secrets |
| **Key management** | Key zeroization deadline, key isolation between domains, key never appears on output ports |
| **Entropy** | RNG output changes each cycle, no stuck-at patterns, minimum entropy per output |
| **Fault detection** | Redundancy check consistency, error flag assertion on fault, safe-state entry on error |
| **Protocol** | Handshake correctness, no data output before authentication, nonce uniqueness |
| **FIPS-specific** | KAT self-test pass, integrity check at power-on, approved algorithm usage |

Properties are parameterized and instantiated by the crypto verification suite based on the project configuration.
